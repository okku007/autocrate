## What the app is

Native macOS menu-bar app. Reads the now-playing Apple Music track as a *seed*,
looks up its BPM + Camelot key (GetSongBPM), and shows a ranked shortlist of
harmonically/tempo-compatible tracks to play next. Show-only (no auto-queue).
Electronic-music focused (genre allowlist). Full intent in
`docs/superpowers/specs/2026-06-24-autocrate-v1-design.md`.

## CURRENT PROBLEM — "still not working" (no shortlist). Prioritized hypotheses.

Evidence after the fix + a fresh run: cache has **exactly 1 row** (the seed only).
That means the library candidate pool was empty (nothing to hydrate). Work these in order:

### H1 (MOST LIKELY): `LibraryReader.readAll()` returns `[]`
`Sources/AutocrateAppKit/Music/LibraryReader.swift` reads the library with plain KVC:
`music.value(forKey: "tracks") as? [NSObject]`. **ScriptingBridge collections are not
reliably reachable via plain KVC** — `tracks` is an `SBElementArray`, usually accessed
through the *typed* generated bridge, not `value(forKey:)`. If this returns nil/empty,
the pool is always empty → only `discover` could ever surface anything → effectively no results.
- **Test:** add a temporary `os_log`/`print` of `library.readAll().count` in `MatchEngine.refresh()`
  (or a one-off), run, check Console. If 0 → confirmed.
- **Fix direction:** generate the typed header (`sdef /System/Applications/Music.app | sdp -fh
  --basename Music`), or use `SBApplication`'s typed `tracks()` element / `SBElementArray`.
  The same KVC concern applies to `revealInMusic` (it iterates `value(forKey:"tracks")`).
- **Note:** `NowPlayingReader` (also KVC, but on a single object `currentTrack` + `playerState`)
  DOES work — proven by the seed reading correctly. The problem is specifically the *collection*.

### H2: Genre allowlist / non-electronic seed → legit `noMatches`
Seed tested was `Hotel California` (Eagles, rock, 75 BPM). The pipeline only keeps candidates
whose genre ∈ `{dance,electronic,house,techno,trance,dubstep,bass,electronica}` AND within ±6%
of 75 BPM. Almost nothing electronic sits at 75 BPM → `noMatches` is *correct* here.
- **Test:** seed with an actual electronic track (e.g. play deadmau5 – Strobe, 128 BPM) and
  see if matches appear once H1 is fixed.
- Also check the user's library genre tags actually use these umbrella names (Apple tags vary,
  e.g. "Dance" vs "Electronic" vs subgenres). Allowlist is hardcoded in
  `CandidatePipeline.allowlist` and mirrored in `MatchEngine.refresh()`'s pool filter.

### H3: Lazy-hydration latency / `indexing` UX makes it look dead
`MatchEngine.refresh()` runs the WHOLE pipeline in one `await` before publishing a final
`PanelState` (no incremental updates). On a cold cache it hydrates up to 50 tracks at
~1 req/s (`FeatureHydrator` cap=50, throttle=1s) = **up to ~50s of "loading"** per panel open,
then `discover` adds more network + one iTunes resolve per discover candidate. The panel can
look frozen. Warming the whole library takes many panel-opens (50 new/ open).
- **Consider:** publish intermediate states (show seed immediately; stream matches as they
  hydrate), and/or lower throttle, and/or show a spinner. This is a UX/architecture issue,
  not a crash.

### H4: `discover` endpoints unverified
`GetSongBpmClient.discover` calls `https://api.getsong.co/tempo/?api_key=…&bpm=…` and `…/key/?…&key=…`.
**Only `/search/` was verified live.** The `/tempo/` and `/key/` response shapes (and the
`bpm`/`key` param names) are guesses — decode may silently yield `[]`.
- **Test:** curl both endpoints with the key and inspect JSON; fix `DiscoverResponse`/`DiscoverItem`
  decoding + param names to match. Until then discover likely returns nothing.

### H5: GetSongBPM search misses on `feat.`/`&` titles
3 earlier cache rows were `miss` for `MK – Zone (feat. …)`, `Chris Lake & Abel Balder`, etc.
The lookup sends the literal `song:TITLE artist:ARTIST`; punctuation/featured-artist noise
reduces hits. Lower impact than H1–H2. Consider normalizing the query (strip `(feat. …)`,
split on `&`, try primary artist).

## Critical gotchas

- **Cache has NO TTL** (`FeatureCache`). Any change to parsing/lookup leaves stale rows that
  are never re-fetched. After code changes affecting features, **clear the cache**:
  ```sh
  rm -f "$HOME/Library/Application Support/Autocrate/features.sqlite"*
  ```
  Inspect it with: `sqlite3 -header "$HOME/Library/Application Support/Autocrate/features.sqlite" "SELECT * FROM feature_cache;"`
  - Consider a real fix: store a parser/schema version and invalidate, or don't treat
    `found`+`camelot==null` as terminal.
- **Binary staleness:** the user runs the Xcode build, not `.xcbuild/`. After a source change
  they must rebuild (⌘R). Don't trust a stale `.app`.
- **`MatchEngine` duplicates the allowlist** (its pool filter + `CandidatePipeline.allowlist`).
  Keep them in sync or refactor to one source.
- **`refresh()` only runs once** per panel open (`.task` in `MenuPanelView`). No auto-refresh
  when the track changes. Expected for v1 but relevant when testing.
